# 拆分评测机与主服务设计

## 背景

当前项目是一个 demo 级别的 Scratch 判题系统，所有功能（题目管理、判题、前端）耦合在一个 Express 应用中。数据存储使用 JSON 文件，无并发安全，无提交记录，不适合生产使用。

本次重构将系统拆分为两个独立项目：
- **评测机**（当前项目 `scratch-judge-system`）：纯判题服务，无状态
- **主服务**（新项目 `scratch-oj-server`）：题目管理 + 前端 + 调用评测机

## 架构总览

```
┌─────────────────┐         HTTP API         ┌─────────────────┐
│                 │  ──────────────────────>  │                 │
│   主服务         │    POST /api/judge        │   评测机         │
│   (scratch-oj-  │  <──────────────────────  │   (scratch-     │
│    server)      │    { verdict, details }   │    judge-       │
│                 │                           │    system)      │
└────────┬────────┘                           └─────────────────┘
         │
         ▼
   ┌──────────┐
   │  SQLite  │
   └──────────┘
```

通信方式：HTTP API，主服务作为客户端调用评测机。

## 一、评测机（当前项目改造）

### 1.1 定位

纯判题引擎，无状态，不存储任何题目数据。接收 sb3 文件 + 题目配置 JSON，返回判题结果。

### 1.2 API 接口

#### POST /api/judge

判题接口。

**请求**：`multipart/form-data`

| 字段 | 类型 | 说明 |
|------|------|------|
| file | File | sb3 文件 |
| config | String (JSON) | 题目配置，结构见下 |

**config 结构**：

```json
{
  "testCases": [
    { "input": { "a": 3, "b": 5 }, "output": { "ans": 8 } }
  ],
  "timeLimit": 5000,
  "stepLimit": 100000
}
```

**响应**：

```json
{
  "verdict": "AC",
  "details": [
    {
      "case": 1,
      "verdict": "AC",
      "time": 120,
      "steps": 450
    },
    {
      "case": 2,
      "verdict": "WA",
      "time": 80,
      "steps": 320,
      "expected": { "ans": 10 },
      "actual": { "ans": 15 }
    }
  ],
  "totalTime": 200,
  "totalSteps": 770
}
```

#### GET /api/health

健康检查。

**响应**：

```json
{
  "status": "ok",
  "version": "1.0.0",
  "uptime": 3600
}
```

### 1.3 目录结构

```
scratch-judge-system/
├── src/
│   ├── judge.js               # 核心判题器（VM 生命周期、执行监控）
│   ├── sb3Parser.js           # SB3 解压、变量查找/修改、首次赋值替换
│   ├── verdict.js             # 判定常量 AC/WA/TLE/MLE/RE
│   └── routes/
│       └── judge.js           # 判题路由
├── config/
│   └── default.js             # 配置（端口、默认限制）
├── test/
│   ├── test.judge.js          # 判题集成测试
│   └── test.replaceFirstAssignments.js  # 变量替换单元测试
├── server.js                  # Express 入口
├── package.json
└── CLAUDE.md
```

### 1.4 关键改动

1. **移除题目管理代码**：删除 `src/routes/problems.js`、`problems/` 目录、图片上传逻辑、静态文件服务
2. **简化 server.js**：只保留判题路由和健康检查
3. **配置通过请求传入**：`timeLimit`/`stepLimit`/`testCases` 从请求的 config 字段获取，不再从本地 JSON 读取
4. **增强健康检查**：返回版本号和运行时间
5. **保留核心逻辑**：`judge.js`、`sb3Parser.js`、`verdict.js` 基本不变

### 1.5 配置文件

```js
// config/default.js
module.exports = {
  port: parseInt(process.env.PORT, 10) || 3001,
  defaultTimeLimit: 10000,
  defaultStepLimit: 500000,
  defaultMemoryLimit: 256
};
```

## 二、主服务（新项目）

### 2.1 定位

题目管理 + 前端页面 + 判题代理。存储题目数据，管理图片资源，转发判题请求到评测机。

### 2.2 数据库设计（SQLite）

使用 `better-sqlite3` 作为驱动。

```sql
-- 题目表
CREATE TABLE IF NOT EXISTS problems (
  id TEXT PRIMARY KEY,                -- 唯一标识
  name TEXT NOT NULL,                 -- 题目名称
  description TEXT,                   -- 简短描述（列表展示用）
  content TEXT,                       -- 完整题面（Markdown）
  category TEXT,                      -- 分类：GESP-1 ~ GESP-8、mock
  difficulty TEXT,                    -- easy / medium / hard
  source TEXT,                        -- 真题 / 模拟题 / 自编
  tags TEXT,                          -- JSON 数组存储，如 '["循环","列表"]'
  time_limit INTEGER DEFAULT 5000,    -- 时间限制（ms）
  step_limit INTEGER DEFAULT 100000,  -- 步数限制
  test_cases TEXT NOT NULL,           -- JSON 数组存储
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 图片资源表
CREATE TABLE IF NOT EXISTS assets (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  problem_id TEXT NOT NULL,
  filename TEXT NOT NULL,
  original_name TEXT NOT NULL,
  size INTEGER,
  mime_type TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (problem_id) REFERENCES problems(id) ON DELETE CASCADE
);

-- 索引
CREATE INDEX IF NOT EXISTS idx_problems_category ON problems(category);
CREATE INDEX IF NOT EXISTS idx_problems_difficulty ON problems(difficulty);
CREATE INDEX IF NOT EXISTS idx_problems_source ON problems(source);
CREATE INDEX IF NOT EXISTS idx_assets_problem_id ON assets(problem_id);
```

### 2.3 API 接口

#### 题目管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/problems | 列表，支持筛选参数 |
| GET | /api/problems/:id | 详情 |
| POST | /api/problems | 创建 |
| PUT | /api/problems/:id | 更新 |
| DELETE | /api/problems/:id | 删除（级联删除图片） |

筛选参数：`?category=GESP-1&difficulty=easy&source=真题&tag=循环`

#### 图片资源

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/problems/:id/assets | 上传图片 |
| GET | /api/problems/:id/assets | 获取图片列表 |
| DELETE | /api/problems/:id/assets/:filename | 删除图片 |
| GET | /api/assets/:id/:filename | 访问图片文件（静态服务） |

#### 判题代理

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/judge | 转发到评测机 |

请求格式与当前相同（multipart/form-data: file + problemId），主服务从数据库读取题目配置，组装为评测机所需的 config 字段，转发请求。

#### 健康检查

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/health | 返回服务状态 + 评测机连通性 |


### 2.6 判题代理流程

```
客户端 POST /api/judge (file + problemId)
  │
  ▼
主服务接收文件
  │
  ▼
从 SQLite 查询 problemId 对应的 testCases、timeLimit、stepLimit
  │
  ▼
组装 multipart 请求：
  - file: sb3 文件（原样转发）
  - config: JSON 字符串 { testCases, timeLimit, stepLimit }
  │
  ▼
POST 到评测机 /api/judge
  │
  ▼
将评测机响应原样返回给客户端
```

### 2.7 关键设计点

1. **评测机地址可配置**：通过环境变量 `JUDGE_URL` 或配置文件设定
2. **前端迁移**：当前 `public/index.html` 迁移过来，改造为调用新 API（数据库驱动）
3. **图片存储**：文件存磁盘 `uploads/{problemId}/`，元数据存 SQLite `assets` 表
4. **错误处理**：评测机不可达时返回友好错误信息
5. **健康检查聚合**：主服务 `/api/health` 同时检查自身和评测机状态
